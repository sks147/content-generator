# Plan: Move `mcp_client` into `shared/` as a subpath-exported module

## Context

`ts_root/mcp_client/` is a standalone package holding one small class (`McpClient`, ~25 lines) that only `backend` consumes. Meanwhile `ts_root/shared/` was scaffolded in commit `02795ef` and never filled — `src/index.ts` and `src/tsconfig.json` are both **0 bytes**, and `src/{classes,constants,errors,types}/` are empty (untracked, since git can't track empty dirs). There is no `shared/package.json`, so `shared/` is not currently a resolvable package and nothing in the repo references it.

This plan folds the MCP client into `shared/`, deletes `mcp_client/`, and bootstraps `shared/` into a real package — sized for what it needs to become: **shared types/interfaces between frontend and backend**, plus **backend-only shared code** (MCP now, db handling later).

The driving constraint is that these two audiences have incompatible runtime needs. The frontend is a browser app (React/TanStack/Vite, zero Node deps); `@modelcontextprotocol/sdk` is Node-only. A single flat barrel exporting everything would drag the MCP SDK and `node:` builtins into the Vite bundle on any value import. So the package is split by **runtime boundary**, addressed via subpath exports — which is also what makes `import * as mcp from 'shared/mcp'` work.

## Structure

The empty skeleton's flat by-kind layout (`classes/`, `constants/`, `errors/`, `types/`) can't produce a `shared/mcp` subpath — a subpath needs a directory. Resolution: **domain-first at the top, by-kind inside each domain**. The kind-based grouping is preserved where it carries weight.

```
ts_root/shared/
├── package.json          NEW — name "shared", exports map: "." and "./mcp"
├── package-lock.json     NEW — generated by npm install
├── tsconfig.json         NEW — moved out of src/, composite + declaration + declarationMap
└── src/
    ├── index.ts          → 'shared'       BROWSER-SAFE: cross-app contracts only
    ├── types/
    │   └── poem.ts         Poem, PoemInput
    └── mcp/              → 'shared/mcp'   NODE-ONLY
        ├── index.ts        barrel for the subpath
        ├── classes/
        │   └── mcpClient.ts
        ├── constants/
        │   └── mcp.ts
        ├── errors/
        │   └── mcpError.ts
        └── types/
            └── mcp.ts
```

**The rule that keeps this honest:** the root barrel (`shared`) may only export things that run in a browser — types, interfaces, pure constants. Anything with a runtime dependency gets its own subpath directory and its own `exports` entry. Future `shared/db/` → `'shared/db'` follows the same shape.

`src/tsconfig.json` (0 bytes, misplaced) is deleted; the real tsconfig lands at `shared/tsconfig.json` like every sibling package. The empty `src/classes`, `src/constants`, `src/errors` at root are dropped — they're untracked and would vanish on a fresh clone anyway. They get recreated when there's browser-safe content to put in them.

## Conventions to follow (verified against the codebase)

- Tabs, single quotes, semicolons, trailing commas
- camelCase filenames (`mcpClient.ts`, not `McpClient.ts`) — matches `poemController.ts`, `healthRoute.ts`
- Relative imports carry the `.ts` extension (`rewriteRelativeImportExtensions` rewrites to `.js` on emit)
- **Named re-exports only — there are zero `export *` in the entire codebase.** Barrels list each symbol.
- package.json boilerplate: `"version": "0.1.0"`, `"license": "ISC"`, `"type": "module"`, devDeps pinned exactly (`@types/node` `26.1.1`, `typescript` `7.0.2`)

## Change set

### 1. Create the `shared` package

- `shared/package.json` — name `shared`, dep `@modelcontextprotocol/sdk` `^1.29.0`, `exports` map with `"."` and `"./mcp"` (each with a `types` condition first), `main: "dist/index.js"` retained for convention.
- `shared/tsconfig.json` — copy of `mcp_client/tsconfig.json` verbatim (`composite`/`declaration`/`declarationMap` are load-bearing for project references).
- Delete the 0-byte `shared/src/tsconfig.json`.
- No `shared/.gitignore` needed — the repo-root `.gitignore` already covers `node_modules/`, `dist/`, and `*.tsbuildinfo` globally. (`mcp_client/.gitignore` was a 0-byte placeholder and goes away with the package.)

### 2. Move and split the MCP code

`mcp_client/src/client.ts` splits along the by-kind boundary:

- `mcp/classes/mcpClient.ts` — the `McpClient` class
- `mcp/constants/mcp.ts` — `MCP_CLIENT_NAME`, `MCP_CLIENT_VERSION` (currently hardcoded `'content-generator-client'` / `'0.1.0'` inside the class), `MCP_TOOLS`
- `mcp/types/mcp.ts` — the tool-result content shape (currently an inline `as Array<{type: string; text: string}>` cast)
- `mcp/errors/mcpError.ts` — typed failures replacing the current silent `?? ''` fallback (satisfies CLAUDE.md "errors and edge cases handled")
- `mcp/index.ts` — named re-exports of all of the above

**This is a deliberate behavior change, not a pure move.** Today `callTool` swallows an empty/malformed tool result and returns `''`, so a failed generation surfaces to the API as a poem with empty content — and gets persisted that way. With typed errors it throws instead, and `poemController.createPoem` must catch and return 502 rather than writing an empty row. Worth doing (it's a real bug), but it means `POST /api/poems` gains a failure path that didn't exist before.

### 3. Hoist the FE/BE contract

`Poem` / `PoemInput` are declared but **not exported** in `backend/src/db/poems.ts:3-13`. Move them to `shared/src/types/poem.ts` and re-export from the root barrel; `backend/src/db/poems.ts` imports them via `import type { Poem, PoemInput } from 'shared'`.

This gives the root barrel a real first citizen and is the exact FE/BE contract requirement. **Caveat to decide at implementation time:** `created_at: Date` is what postgres.js returns, but it serializes to a `string` over JSON — so the frontend's view of a `Poem` genuinely differs from the backend's. Options: keep `Date` and let the FE define its own wire type, or model `created_at` as `string` in the shared type and have the backend convert. Flagging rather than silently picking.

### 4. Rewire backend

| File | Change |
|---|---|
| `backend/package.json` | `"mcp-client": "file:../mcp_client"` → `"shared": "file:../shared"` |
| `backend/tsconfig.json` | `references` → `[{ "path": "../shared" }]`; **delete the `paths` block** |
| `backend/src/mcp/poems.ts:1` | `import { McpClient } from 'mcp-client'` → `import * as mcp from 'shared/mcp'` |
| `backend/src/db/poems.ts` | local `Poem`/`PoemInput` → `import type { ... } from 'shared'` |

The `paths` block goes away because it was measured to be redundant: deleting it + both `dist/` dirs and running `npx tsc --build` from backend exits clean, since TS resolves through the npm `file:` symlink → `package.json` → `dist/*.d.ts`. With subpaths, keeping it would also mean maintaining the subpath list in two places instead of letting the `exports` map be the single source of truth.

**Honest caveat:** that test covered a *bare* specifier (`mcp-client`). Resolving a *subpath* (`shared/mcp`) goes through the `exports` map, which is a different code path — and the map points at `dist/`, which doesn't exist on a cold clone. Verification step 1 tests exactly this cold-start case. If it fails, the fallback is one line: restore `paths` with `"shared"` and `"shared/mcp"` entries pointing at source. Nothing else in the plan depends on the outcome.

### 5. Rewire Docker

**Verified constraint:** `backend/node_modules` contains **no** `@modelcontextprotocol` — only `mcp_client/node_modules` does. npm does not install a `file:`-linked package's dependencies into the consumer; Node resolves the symlink to its real path and loads deps from the linked package's own `node_modules`. **The linked package needs its own `node_modules` in the image** — that's why the two-WORKDIR pattern exists, and it must carry over.

- `backend/Dockerfile` — `/mcp_client` → `/shared`, `COPY mcp_client/*` → `COPY shared/*`
- `backend/docker-entrypoint.sh:9-12` — `/mcp_client` → `/shared`
- `infra/docker-compose.yml` — backend volume mounts `../ts_root/mcp_client/src:/mcp_client/src` → `../ts_root/shared/src:/shared/src` (same for tsconfig); volume `mcp_client_node_modules` → `shared_node_modules` in both the service block and the top-level `volumes:` block

**Careful:** `mcp-server` / `mcp_servers` / `MCP_SERVER_URL` are a *different* package. A find/replace on `mcp` must not touch them.

### 6. Fix `backend/Dockerfile_prod` (pre-existing bug, surfaced by this work)

Broken today, before any of this: `docker-compose-prod.yml:35` sets context to `../ts_root/backend`, but `backend/package.json` declares `file:../mcp_client` — a path outside that context — so `RUN npm ci` cannot resolve it. It also runs `npx tsc` rather than `tsc --build`, so it would never build the referenced project. The prod image has almost certainly not built since the MCP dep was added.

Fix: widen `docker-compose-prod.yml` backend context to `../ts_root` with `dockerfile: backend/Dockerfile_prod`, and restructure `Dockerfile_prod` to mirror the dev two-WORKDIR pattern, using `tsc --build` and carrying `/shared/dist` + `/shared/node_modules` into the runtime stage.

### 7. Delete `mcp_client/` and regenerate

- `git rm -r ts_root/mcp_client` (tracked: `.gitignore`, `package.json`, `package-lock.json`, `src/client.ts`, `src/index.ts`)
- `npm install` in `shared/` (generates `package-lock.json`) and in `backend/` (rewrites the 3 stale `mcp-client` lock entries, re-points the now-dangling `backend/node_modules/mcp-client` symlink)
- Stale gitignored artifacts regenerate on next build: `backend/tsconfig.tsbuildinfo` (embeds `../mcp_client/dist/*` paths), `backend/dist/`, `mcp_client/dist/`

`plans/00001-basic-mcp-setup.md` references `mcp_client` throughout but is a historical record of what was built — leave it alone.

## Deliberately out of scope (natural follow-ups)

- **Wire `mcp_servers` to `shared/mcp`.** The tool name `'generate_poem'` and its arg shape are a contract between `mcp_servers` (which registers the tool) and `backend` (which calls it), currently duplicated as string literals on both sides. Once `MCP_TOOLS` lives in `shared/mcp`, having both consume it would make the contract single-source. Needs the same `file:` + `references` wiring backend gets here.
- **Wire the frontend to `shared`.** Confirmed viable — frontend's `moduleResolution: "bundler"` honors `exports` maps, so `import type { Poem } from 'shared'` will resolve to the root barrel. Not done now because there's no frontend code consuming it yet; the structure is what makes it a later one-liner.
- **`ts_root/.dockerignore` is missing.** The dev backend build context is `../ts_root`, so `backend/.dockerignore` does not apply to it — the entire tree, including every `node_modules/`, uploads as build context on each build. Pre-existing and unrelated, but widening the prod context to `../ts_root` (step 6) inherits the same problem.
